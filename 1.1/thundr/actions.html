thundr Actions

Routes

Incoming web requests are handled by thundr using routes.

A route is a url wildcard pattern which maps to an action. Each route is of a http request type - GET, PUT, POST or DELETE.

Routes are defined in Modules at application startup. You register your routes on the Routes instance.

{% highlight java %} public class ApplicationModule implements Module { @Override public void start(UpdatableInjectionContext injectionContext) { Routes routes = injectionContext.get(Routes.class); routes.addRoute(RouteType.GET, "/", "dashboard", new MethodAction(AccountController.class, "dashboard")); routes.addRoute(RouteType.GET, "/signup", "signup.get", new MethodAction(AccountController.class, "signupGet")); routes.addRoute(RouteType.POST, "/signup", "signup.post", new MethodAction(AccountController.class, "signupPost")); routes.addRoute(RouteType.GET, "/view/{id}", "signup.post", new MethodAction(AccountController.class, "view")); } {% endhighlight %}

Routes can map to different types of actions. The most common action is MethodAction, which will result in a method being invoked on a controller.

In the above example, requests to the base url '/' will be processed by the dashboard method defined in the AccountController class. GET Requests on '/signup/' will be processed by signupGet() and POSTs will be processed by signupPost(). Finally, requests matching the path /view/* will be processed by the method with the name view, with the addition that the value matching the wildcard can be passed as a parameter with the name 'id', permitting view(String id).

The path defined in a route is used as a wildcard match, and the action invoked is the first matched route based on the order they're added.

The following are valid in a route path:

  • /path/components - standard path components
  • /path/{variables}/ - path variables are named parameters available when binding to controller methods
  • /path/*/wildcards - wildcards with a single asterisk match just one path fragment. i.e. /path/value/wildcards matches and /path/multi/value/wildcards does not
  • /path/**/wildcards - double wildcards match any number of path fragments. i.e. both /path/value/wildcards and /path/multi/value/wildcards match
  • /file/*.ext - extensions can be used in conjuction with other matching rules

Actions

Each route maps to a single Action. Actions determine how your application handles a web request. In general, an action will usually result in some code being invoked, but other types of actions exist.

Currently thundr supports the following actions out of the box:

MethodAction

This type of action results in a method being invoked on a class. Classes invoked like this are referred to as controllers, however this is a semantic distinction - any class can be used as a controller. You can read more about controllers here.

For example, the route

{% highlight java %} ... routes.addRoute(RouteType.GET, "/route/{variable}", null, new MethodAction(Controller.class, "actionMethod")); ... {% endhighlight %}

will invoke the method

{% highlight java %} package com.threewks.thundr; public class Controller { ... public StringView actionMethod(String variable){ doSomething(); return new StringView("Ok!"); } ... {% endhighlight %}

StaticResourceAction

This type of action results in a resource being served directly. This is useful when serving javascript, css, text files or other resources which don't require any processing by your application.

The routes below result in the favicon and all files under the folder 'static' being server as static resources.

{% highlight java %} ... routes.addRoute(RouteType.GET, "/favicon.ico", null, new StaticResourceAction()); routes.addRoute(RouteType.GET, "/static/**", null, new StaticResourceAction()); ... {% endhighlight %}

RedirectAction

This type of action results in a HTTP redirect being sent to the web request client (i.e. the user's browser). This will result in the browser requesting the resource specified in the redirect route.
The redirect is a temporary redirect, or 302.

{% highlight java %} ... routes.addRoute(RouteType.GET, "/old", null, new RedirectAction("/new")); routes.addRoute(RouteType.GET, "/old/{withParam}", null, new RedirectAction("/new/{withParam}")); ... {% endhighlight %}

RewriteAction

This type of action is used to cause another route to handle the request. It is similar in concept to the rewrite rules in an Apache config, but could also be considered to act as an alias.

In the below example, the rewrite rule will cause the rewritten route to be invoked as though the request client invoked it directly, but unlike a redirect will be served with the url it was requested on.

{% highlight java %} ... routes.addRoute(RouteType.GET, "/requestedPath", null, new RewriteAction("/rewritten")); routes.addRoute(RouteType.GET, "/rewritten", null, new MethodAction(Controller.class, "method")); ... {% endhighlight %}

ActionResolver Advanced

Each type of action is handled by an ActionResolver. ActionResolvers process an Action when a route is requested. You can create and configure custom Action and ActionResolvers to handle routes, although in general this is not necessary.

An action resolver is responsible for resolving an action when a route is requested by an http client

Action resolution usually involves binding data based on route configuration and request parameters, and serving up response content and http response codes. This is done using the standard Java Servlet interfaces, HttpServletRequest and HttpServletResponse.

If you wish to register a custom Action and ActionResolver, do so when your Module is configured.

{% highlight java %} public class MyModule implements Module { @Override public void configure(UpdatableInjectionContext injectionContext) { super.configure(injectionContext); Routes routes = injectionContext.get(Routes.class); routes.addActionResolver(MyAction.class, new MyActionResolver()); {% endhighlight %}
